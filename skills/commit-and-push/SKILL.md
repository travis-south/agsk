# Commit and Push

Finalize the current task on the current branch. Account for every changed path, preserve unrelated work, create the smallest coherent commit set, and publish it with a standard fast-forward push.

## 1. Account for the change set

1. Resolve the repository root, current branch, remotes, upstream, and working-tree state.
2. Read repository instructions, commit-message configuration, relevant package scripts, and recent commit subjects.
3. Inspect staged, unstaged, and untracked changes. Use the current task and conversation to classify every path as intended or unrelated.
4. Treat a detached `HEAD`, unresolved merge or rebase, unclear change ownership, or ambiguous push remote as a blocker. Preserve state and request only the missing decision.

Pass this step when the current branch and push target are known and every changed path is classified.

## 2. Prepare and validate

1. Stage intended paths or hunks explicitly. Preserve unrelated staged and unstaged work.
2. Review the complete staged diff and staged file list.
3. Run `git diff --cached --check` and any configured secret, generated-file, or commit-time checks that apply.

Pass this step when the index contains the complete intended change set, contains no unrelated change, and all applicable checks pass.

## Commit subject contract

Use the repository's commitlint rules. Otherwise use:

```text
<type>(<scope>): <terse imperative summary>
```

Add `!` only for a breaking change. Add a body only for required context, evidence, links, or breaking-change details.

Choose `<type>` from the change intent and the repository's allowed types. Choose `<scope>` in this order:

1. **Tracker identifier from the branch.** Prefer the canonical issue key over its descriptive suffix.
   - Use an exact Jira- or Linear-style key matching `[A-Z][A-Z0-9]+-[0-9]+`, such as `BILLING-2819` or `CPA-1411`.
   - Accept another tracker identifier when repository history, linked issue context, or a clearly delimited branch token establishes it, such as `GH-482` or `issue-482`.
   - For an issue-bearing prefix followed by a numeric identifier, such as `issue/482-fix-timeout` or `spec/48-modernize-dependencies`, use the repository's established form. With GitHub or GitLab and no established form, use `#482` or `#48`.
   - Treat a leading number as an issue identifier only when the repository or hosting convention establishes that branch pattern.
2. **Usable branch topic.** Strip established workflow or owner prefixes, retain the concise kebab-case topic, and use it when the branch describes one stable concern.
3. **Changed component.** Use the narrowest stable module, package, or repository scope shared by the staged changes.

Treat default, integration, release-only, autogenerated, and opaque branch names as unusable topics. Preserve tracker casing; normalize descriptive topics to repository rules. Validate the proposed subject with the repository's installed commitlint command when available, without installing new dependencies.

Default examples:

```text
spec/BILLING-2819-retire-e2e  -> chore(BILLING-2819): retire e2e stack
fix/issue/482-timeout         -> fix(#482): handle resolver timeout
feat/cache-warming            -> feat(cache-warming): add cache warmup
main                          -> fix(auth): reject expired sessions
```

## 3. Commit

1. Create one commit when one subject accurately describes the staged set. Split independent concerns into explicit-path or hunk commits in dependency order.
2. Let repository hooks run. When a hook changes files, re-account for those changes, rerun affected checks, and stage only attributable fixes before retrying.
3. Verify every new commit's subject, content, and author-visible diff. Keep existing history intact unless the user explicitly requests history rewriting.

Pass this step when every intended change is included in a valid new commit, every commit satisfies the subject contract, and unrelated work remains outside those commits.

## 4. Push and verify

1. Push to the configured upstream. When no upstream exists and one push remote is unambiguous, set it with `git push -u <remote> HEAD`.
2. Use a standard fast-forward push. Treat a non-fast-forward rejection, failed hook, authentication error, or remote-policy rejection as a blocker with local commits preserved.
3. Verify the upstream tracking ref resolves to the same commit as local `HEAD`.

Pass this step only when every new commit is reachable from the upstream branch at the same `HEAD`.

## 5. Report

Keep the result terse:

- new commit short SHA and subject;
- pushed branch and upstream;
- checks run and outcome.

For a blocker, report the exact failure, preserved state, and retry point.
